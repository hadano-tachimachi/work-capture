# AI構造化の考え方

AIは「判断者」ではない。

AIは **情報整理担当** とする。

役割は以下に限定する。

-   目的
-   背景
-   タスク
-   期限
-   アクションプラン
-   メモ
-   判断事項
-   次の一歩
-   Project候補
-   Context候補

## 禁止事項

AIは入力に存在しない情報を補完しない。

-   推測で人物を追加しない
-   推測で期限を作らない
-   推測で担当者を決めない
-   推測でタスクを増やさない

不明なものは「候補」または「未設定」として扱う。

------------------------------------------------------------------------

# 安心設計

AIの出力をそのままDBへ保存しない。

## 基本フロー

``` text
録音
↓
文字起こし
↓
文字起こし本文を保存
↓
AI構造化
↓
AI結果(raw)保存
↓
プログラムで検証
↓
構造化データ生成
↓
Inbox Zero
↓
人間が確認
↓
正式タスク化
```

------------------------------------------------------------------------

# 保存構造

## 1. work_captures

音声入力1回につき1件。

保存内容

-   audio_url
-   transcript_text
-   status
-   created_at

status例

-   transcribed
-   ai_processed
-   validation_failed
-   ready_for_review
-   confirmed

------------------------------------------------------------------------

## 2. ai_parse_results

AIの出力をそのまま保存。

保存内容

-   raw_output
-   parsed_json
-   validation_status
-   validation_errors
-   model_name
-   prompt_version

目的

AI解析結果の証拠を残す。

------------------------------------------------------------------------

## 3. structured_items

AIが抽出した候補。

type例

-   purpose
-   background
-   task
-   action
-   note
-   decision
-   due_date
-   next_action

まだ正式データではない。

------------------------------------------------------------------------

## 4. tasks

人間が確認後、正式な仕事として登録。

保存内容

-   title
-   description
-   due_date
-   priority
-   status
-   project
-   context
-   assigned_to

# 設計思想

AIは万能ではない。

Work Captureでは

-   元データを残す
-   AI結果は候補
-   プログラムが検証
-   人間が確定

という三段階で品質を担保する。

------------------------------------------------------------------------

# 判断基準

新しい機能を追加する時は必ず問う。

> この機能は「迷う時間」を減らすか？

YESなら採用候補。

NOなら採用しない。

------------------------------------------------------------------------

# 現時点での結論

Work Captureは

「AIがタスクを勝手に決めるシステム」

ではない。

**AIが思考を構造化し、人間が安心して仕事へ変換できるシステム**である。

安心設計の原則は次の一文に集約される。

> **AIは候補を作る。プログラムが検証する。人間が確定する。**
